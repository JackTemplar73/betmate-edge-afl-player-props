# BetMate Edge AFL Player Props: Base44 Full Build Specification

Use this document as the single implementation brief for Base44 so it can reproduce the same AFL player props output currently generated by the Codex `AFLplayerprops` workflow and displayed in the local HTML report.

This document covers:
- the product requirements
- the UI and table behavior
- the Base44 data model
- the Base44 backend refresh function
- the Codex-side JSON refresh API contract
- the integration responsibilities between Base44 and Codex

The goal is for Base44 to become the native app and orchestration layer, while Codex remains the source of truth for model refresh, odds fetch, EV/QI scoring, and prop normalization.

---

## 1. Product Goal

Build a Base44 app called `BetMate Edge AFL Player Props`.

The app must show current-round AFL player props using the same high-value logic and structured output currently produced by Codex.

The app must:
- fetch fresh data on demand
- ingest structured prop rows from Codex
- show a `Round Summary` tab
- show one tab per round match in game start time order
- show a final `Tracking` tab
- split props into `High Value Props` and `Multi Props Only`
- use the same core thresholds currently used in Codex

This app is for `AFL player props only`.

Do not parse the Codex HTML report in Base44.
Base44 must consume structured JSON only.

## 2. Core Business Rules

### Visible row rules

Only show rows that satisfy all of the following:
- `QI >= 80`
- `EV > 0`
- approved actionable market only

Approved actionable markets:
- `Disposals`
- `Tackles`
- `Goals`

Do not include:
- fantasy points
- ranking points
- unbacktested markets
- invented or unavailable markets

### Segment rules

Split visible props into:

- `High Value Props`
  - `QI >= 80`
  - `Price >= 1.80`

- `Multi Props Only`
  - `QI >= 80`
  - `Price < 1.80`

### Sorting rules

All tables on all tabs must be sorted by:
1. `QI` descending
2. `Prob Edge` descending
3. `Player` ascending

### Display rules

- Show team names only where possible
- Do not use mascot names in display headings if a clean team name is available
- `QI` displays as a whole number
- `Model Line` displays rounded to the nearest whole number
- `Model Prob` displays as a percentage
- `Prob Edge` displays as a percentage
- Every visible row must have a populated `Stake`

## 10. Base44 Backend Function

Create a backend function:
- `refreshAflPlayerProps`

This function is the Base44-side orchestrator.

It does not do the scoring itself.

Instead, it calls a Codex refresh endpoint, ingests the JSON response, and updates Base44 entities.

### Base44 secrets

Create these app secrets:
- `CODEX_REFRESH_URL`
- `CODEX_REFRESH_TOKEN`

These secrets must never be exposed in frontend code.

### Backend function responsibilities

The function must:
1. create a `RefreshRun` row with status `RUNNING`
2. call the Codex endpoint `POST /refresh`
3. pass authorization header if `CODEX_REFRESH_TOKEN` exists
4. parse the response
5. if failure:
   - mark `RefreshRun` as `FAILED`
   - update `AppState`
   - return a clean error payload
6. if success:
   - upsert `Match` rows using `gameKey`
   - replace or upsert `Prop` rows using `propKey` for the current round
   - create a successful `RefreshRun`
   - update `AppState`
   - return a compact success payload

### Base44 backend return shape

Return something like:

```json
{
  "ok": true,
  "updatedAt": "2026-05-28 17:11:00",
  "roundLabel": "Round 12",
  "gamesCount": 7,
  "propsCount": 161,
  "message": "Refreshed successfully"
}
```

On error:

```json
{
  "ok": false,
  "message": "HTTP Error 429: Too Many Requests"
}
```

## 14. Codex Endpoint: POST /refresh

### Route

`POST /refresh`

### Purpose

This endpoint must:
1. fetch fresh model-data inputs
2. fetch fresh Odds API events
3. fetch current-round player prop odds
4. rerun scoring
5. rebuild the normalized current-round prop universe
6. return structured JSON for Base44

### Request headers

- `Content-Type: application/json`
- `Authorization: Bearer <CODEX_REFRESH_TOKEN>` optional but recommended

### Request body

```json
{
  "game": null,
  "output": "afl_player_props_stk_haw_walters_report.html",
  "roundLabelOverride": null,
  "minQi": 80,
  "highValueMinPrice": 1.80
}
```

### Request rules

- `game`
  - `null` means full round refresh
  - optional specific match string if Codex also wants to rebuild a match-specific HTML file

- `output`
  - optional HTML output filename for Codex’s own local report build
  - Base44 does not need HTML, but Codex may still produce it

- `roundLabelOverride`
  - optional manual round label if required

- `minQi`
  - default `80`

- `highValueMinPrice`
  - default `1.80`

## 16. Codex-side Filtering Rules

Codex should only return rows that satisfy:
- approved market only
- `QI >= minQi`
- `EV > 0`

Then assign:
- `segment = High Value Props` if `price >= highValueMinPrice`
- `segment = Multi Props Only` if `0 < price < highValueMinPrice`

Do not return fantasy points or other unvalidated markets in actionable output.

---

## 17. Codex-side Sorting Rules

Before returning props, sort them by:
1. `qi` descending
2. `probEdge` descending
3. `player` ascending

---

## 18. Codex-side Status Rules

Codex should infer prop status from current time versus `commenceTime`:

- `UPCOMING`
  - current time is before game start

- `IN_PLAY`
  - current time is after game start
  - final result not yet available

- `COMPLETED`
  - final result available

### Result field rules

- blank if `UPCOMING`
- `IN PLAY` if match has started but not settled
- `WIN`, `LOSS`, or `PUSH` if settled

## 19. Codex Response Field Definitions

### games[]

- `gameKey`
  - stable unique match identifier
  - recommended pattern: `short_home_v_short_away_yyyy_mm_dd`

- `displayName`
  - clean team-only display name
  - example: `St Kilda v Hawthorn`

- `sortOrder`
  - start-time order for the round

- `hasLiveBooks`
  - boolean indicating whether player prop books were returned

- `bookmakersCount`
  - count of books returned for requested player prop markets

### props[]

- `propKey`
  - stable unique row key
  - recommended to combine game, player, market, side, line, and bookie

- `roundLabel`
  - example: `Round 12`

- `team`
  - team name only

- `market`
  - `Disposals`, `Tackles`, or `Goals`

- `side`
  - `Over` or `Under`

- `modelLine`
  - raw numeric model line

- `marketLine`
  - raw numeric market line

- `modelProb`
  - decimal probability like `0.74`

- `price`
  - decimal odds like `1.87`

- `bookie`
  - selected bookmaker name

- `probEdge`
  - decimal probability edge like `0.21`

- `stake`
  - numeric stake units

- `qi`
  - numeric QI score

- `segment`
  - `High Value Props` or `Multi Props Only`

- `sourceSelection`
  - can use values like:
    - `OPEN`
    - `TRACKED`
    - `PORTFOLIO_BET`

## 20. Base44 Ingestion Logic

Base44 backend function `refreshAflPlayerProps` should:

1. optionally call `GET /health`
2. call `POST /refresh`
3. parse the response
4. upsert `Match` rows by `gameKey`
5. replace or upsert `Prop` rows by `propKey` for the active round
6. create a `RefreshRun` row
7. update `AppState`

### Upsert guidance

Use:
- `gameKey` as the stable match identifier
- `propKey` as the stable prop identifier

If the active round is refreshed again:
- update existing rows where keys match
- add new rows that appear
- remove or archive rows for the current round that no longer exist in the latest payload, depending on Base44 implementation preference

Recommended default:
- replace current-round props with the latest returned current-round dataset

## 21. Error Handling Requirements

Handle these clearly in Base44:

- no refresh has ever run
- refresh service unavailable
- refresh failed
- no live books returned
- no qualifying props for a match
- refresh already running

Handle these clearly in Codex:

- missing API key
- rate limit / quota hit
- malformed upstream payload
- in-progress refresh lock
